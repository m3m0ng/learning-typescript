Lesson 2 of your TypeScript journey

TypeScript Through a Python Lens

Goal: re-anchor what you saw in Lesson 1 using Python (which you already know) and a robotics example, then add one new idea: lists of structured readings.

Forget JavaScript for this lesson. Every TypeScript idea below maps onto something you've already done in Python with type hints and dataclasses, or with sensor data on Arduino.

1. Variables: Python type hints vs. TypeScript annotations

In Python, you can optionally write a type hint after a variable name. TypeScript makes this mandatory-feeling (and enforced) using a colon, the same way:

Python (type hint)

battery_voltage: float = 12.6
motor_count: int = 4
robot_name: str = "Rover-1"
is_armed: bool = True

TypeScript

let batteryVoltage: number = 12.6;
let motorCount: number = 4;
let robotName: string = "Rover-1";
let isArmed: boolean = true;

The only real differences: let instead of nothing (TypeScript needs a keyword to declare a variable), every line ends with ;, and Python's int and float both become TypeScript's single number type. Otherwise — same idea, same intent: "this name will only ever hold this kind of value."

Quiz: in Python you'd write temperature: float = 23.5. What's the TypeScript equivalent?

2. Functions: type hints on parameters and return values

You've done this in Python too — annotating what a function expects and what it gives back. Here's a function that converts a sensor's raw reading (0-1023, like an Arduino analog pin) into a voltage:

Python

def raw_to_voltage(raw: int) -> float:
    return (raw / 1023) * 5.0

TypeScript

function rawToVoltage(raw: number): number {
  return (raw / 1023) * 5.0;
}

Same shape: (parameter: type) -> return_type in Python becomes (parameter: type): return_type in TypeScript. If you tried to call rawToVoltage("high"), TypeScript stops you immediately — just like a Python type checker (e.g. mypy) would flag raw_to_voltage("high").

3. Describing a "reading": Python dataclass vs. TypeScript interface

Imagine you're logging readings from a robot's sensor array — a timestamp, a sensor name, and a value. In Python you might define a dataclass to describe that shape:

Python

from dataclasses import dataclass

@dataclass
class SensorReading:
    timestamp: int
    sensor: str
    value: float

TypeScript

interface SensorReading {
  timestamp: number;
  sensor: string;
  value: number;
}
A Python dataclass and a TypeScript interface do almost the same job: they describe the shape of a record — what fields it has and what type each one is. The dataclass actually creates an object you instantiate; the interface is purely a "blueprint" the type checker uses — it doesn't exist at runtime at all. But for reading code, treat them as the same idea: "here's what this kind of record looks like."

Quiz: which best describes what a TypeScript interface is for?

4. New idea: a list of readings (array of objects)

On a real robot, you don't get one reading — you get a stream of them, like a Python list[SensorReading]. TypeScript writes "a list of X" as X[]:

Python

readings: list[SensorReading] = [
    SensorReading(timestamp=1001, sensor="temp", value=23.5),
    SensorReading(timestamp=1002, sensor="temp", value=23.7),
]

TypeScript

const readings: SensorReading[] = [
  { timestamp: 1001, sensor: "temp", value: 23.5 },
  { timestamp: 1002, sensor: "temp", value: 23.7 },
];

SensorReading[] reads as "an array where every item must match the SensorReading shape." If you tried to sneak in { timestamp: "oops", sensor: "temp", value: 23.5 } — a string where a number should be — TypeScript would flag that one entry, the same way a type checker would flag a wrong type inside a Python list.

5. Putting it together: a function over the list

Here's a function that takes the whole list of readings and returns just the average value — something you'd genuinely do when processing logged sensor data: 

function averageValue(readings: SensorReading[]): number {
  const total = readings.reduce((sum, r) => sum + r.value, 0);
  return total / readings.length;
}

Reading this with what you know: readings: SensorReading[] — "give me a list of sensor readings." : number — "I'll give you back a single number (the average)." The body is doing roughly what sum(r.value for r in readings) / len(readings) would do in Python — don't worry about reduce syntax yet, just recognize the types in and out.

Try it yourself

In the TypeScript Playground, paste the SensorReading interface, the readings array, and the averageValue function from above. Then add this line at the bottom and click Run: 

console.log(averageValue(readings));

Then try breaking it on purpose: change one value: 23.5 to value: "23.5" (a string). You won't even need to click Run — TypeScript will underline the problem immediately. Hover over the underline to read the error message; it should sound a lot like a Python type-checker complaint.
