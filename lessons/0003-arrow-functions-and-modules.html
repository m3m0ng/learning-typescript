Lesson 3 of your TypeScript journey

Arrow Functions, Modules, and Reading Real Files

Goal: recognize the two patterns that show up at the top and throughout almost every real .ts file — import/export and arrow functions — so a real file stops looking like noise.

New direction from here on: examples will come from CLI tools, scripts, and small TUI/agent-style code — the kind of TypeScript that AI assistants actually generate — instead of robotics. Python is still your anchor for syntax intuition, just not the example domain anymore.

1. The big picture: what survives compilation?

Quick recap, because it matters for everything below: TypeScript is JavaScript plus types. Before your code runs (in a browser, in Node, anywhere), a compiler called tsc strips out every type annotation and produces plain JavaScript. That JS is what actually executes.

From this lesson on, code samples are tagged so you always know which bucket something is in:

JS — real syntax, exists at runtime, would still be there in the compiled output (functions, import/export, .map(), if, loops, etc.)

TS — type-only, vanishes when compiled (: number, interface, generic <T> parameters, etc.)

Why this matters for your goal: when you're reading a TUI component or a CLI script, the JS parts are "what the program actually does." The TS parts are "guardrails the editor/compiler enforces." Knowing which is which helps you separate "how does this behave" from "why is the editor complaining."

2. Arrow functions JS — the modern way to write functions

Lesson 2 used the function keyword. In real-world TS (and almost all AI output), the far more common style is an arrow function assigned to a const. Same function, two ways to write it:

Named function (Lesson 2 style)

function double(n: number): number {
  return n * 2;
}

Arrow function (the common style)

const double = (n: number): number => {
  return n * 2;
};

=> ("fat arrow") replaces the function keyword and the name moves to a const on the left. Everything else — parameter types, return type — works exactly the same as Lesson 2.

There's a shorthand too: if the function body is a single expression that you'd just return, you can drop the braces and the return keyword entirely: 

const double = (n: number): number => n * 2;

This isn't quite Python's lambda (which is limited to one expression and can't be reused as easily) — think of it more like: any time you'd write def and assign the function to a name, JS/TS lets you write it as const name = (...) => ... instead. Both forms exist in real code; arrow functions are just far more common.

3. Arrow functions as arguments — .map() and .filter() JS

The single most common place you'll see arrow functions is passed directly into another function — usually an array method. Imagine a small CLI to-do tool: 

interface Task {
  id: number;
  title: string;
  done: boolean;
}

const tasks: Task[] = [
  { id: 1, title: "Write lesson 3", done: true },
  { id: 2, title: "Review PR", done: false },
  { id: 3, title: "Deploy app", done: false },
];

// keep only tasks that aren't done
const incomplete = tasks.filter(task => !task.done);

// pull out just the titles
const titles = tasks.map(task => task.title);

Read tasks.filter(task => !task.done) as: "for each task in tasks, keep it only if !task.done is true." That's the same idea as Python's: 

incomplete = [task for task in tasks if not task.done]
titles = [task.title for task in tasks]
.filter() and .map() are everywhere in AI-generated TS — transforming lists of files, messages, API results, UI items. Whenever you see something.filter(x => ...) or something.map(x => ...), mentally translate it to a Python list comprehension and you've got the gist.

4. Modules: import and export JS

Real projects split code across files. export marks something as usable from other files; import brings it in. This is the very first thing you'll see at the top of almost any .ts file:

types.ts

export interface Task {
  id: number;
  title: string;
  done: boolean;
}

tasks.ts

import { Task } from "./types";

export const getIncomplete = (tasks: Task[]): Task[] =>
  tasks.filter(task => !task.done);

This is the same idea as Python's from types import Task — tasks.ts is saying "give me the Task shape that lives in ./types, so I can use it here." export on the interface and the function means "other files are allowed to import these too." Anything not exported is private to that file — like a name without an __all__ entry, or a "module-private" helper.

5. Putting it together

Here's a small file that uses everything from this lesson — this is realistically close to a snippet an AI assistant might generate for a CLI to-do tool: 

tasks.ts
import { Task } from "./types";

export const getIncomplete = (tasks: Task[]): Task[] =>
  tasks.filter(task => !task.done);

export const taskTitles = (tasks: Task[]): string[] =>
  tasks.map(task => task.title);

Reading it top to bottom: "bring in the Task shape from types.ts; export a function getIncomplete that takes a list of tasks and returns the ones not done; export another function taskTitles that takes a list of tasks and returns just their titles as strings." Two arrow functions, both exported, both typed — a tiny but completely realistic module.

Try it yourself

Open the TypeScript Playground. The Playground runs a single file, so combine everything into one — paste the Task interface, the tasks array, and the getIncomplete / taskTitles functions (skip the import/export lines for now, since there's only one file). Add at the bottom: 

console.log(getIncomplete(tasks));
console.log(taskTitles(tasks));

Then break it on purpose: change task => !task.done to task => !task.donee (typo). TypeScript should immediately underline donee — because it doesn't exist on the Task interface. That's the type checker catching a typo that, in plain JavaScript, would silently return undefined and cause a confusing bug later.
