Lesson 4 of your TypeScript journey

async / await and Promise<T>

Goal: read the code that talks to the outside world — files, APIs, processes — without it looking like magic. This is the single most common pattern in real AI-generated TypeScript, and it's where you'll meet your first generic.

Why this lesson, now: almost anything useful a CLI tool or TUI does — fetch from an API, read a file, run a subprocess — takes time, and JavaScript handles "takes time" with async/await. If you've ever seen AI-generated TS sprinkled with async, await, and Promise<...> and skimmed past it, this lesson makes those three words click.

1. The core idea: a value that isn't ready yet

Some results are available instantly: 2 + 2 is 4 right now. But "the contents of a file" or "the response from an API" can't be handed to you the instant you ask — there's a delay while the disk or network does its work.

JavaScript's answer is a Promise: an object you get back immediately that represents "a value that will arrive later." Think of it as a claim ticket at a coat check — you hand over your coat and get a ticket on the spot. The ticket isn't the coat, but it's your guarantee you can collect the coat once it's ready.

Python analogy: this is exactly async def and await in asyncio. A Python async def function returns a coroutine you await; a JS async function returns a Promise you await. Same shape, same instinct — the keywords are nearly identical.

2. Promise<T> — your first generic TS

Here's where TypeScript adds value. A promise will eventually produce a value — but a value of what type? That's what the <T> says: 

Promise<string>   // "a value arriving later, and it will be a string"
Promise<number>   // "...will be a number"
Promise<Task[]>   // "...will be an array of Task"
Promise<void>     // "...finishes, but produces no useful value"

The angle brackets <...> are a generic: a slot you fill in. Promise is the container; the T inside is "what's in the box once it arrives." You already met this shape without noticing — Array<string> is the same idea (it's just usually written string[]).

Read Promise<T> out loud as "a promise of a T." Promise<string> = "a promise of a string." That phrasing will carry you through nearly every generic you'll ever read.

3. async and await JS

Working with promises directly is clunky. async and await are the syntax that makes "code that waits" read like ordinary top-to-bottom code:

  • async in front of a function means "this function does some waiting, and it returns a Promise."
  • await in front of a promise means "pause here until the value arrives, then give me the unwrapped value."
async function loadTitle(): Promise<string> {
  const response = await fetch("https://api.example.com/title");
  const text: string = await response.text();
  return text;
}

Notice the two halves of the same coin: the function is declared : Promise<string> (from the outside, callers get a promise), but return text; returns a plain string. An async function automatically wraps its return value in a promise — you write return text, callers receive a Promise<string>.

And await response.text() does the reverse: response.text() hands back a Promise<string>, and await unwraps it into the actual string you store in text.

Python (asyncio)

async def load_title() -> str:
    resp = await fetch(url)
    text = await resp.text()
    return text

TypeScript

async function loadTitle(): Promise<string> {
  const resp = await fetch(url);
  const text = await resp.text();
  return text;
}

The one wrinkle worth knowing: Python writes the return type as -> str, but TS writes it as Promise<string> — the promise wrapper is explicit in the type. That's the small mental adjustment: an async function's return type is always Promise<something>, never just something.

4. Arrow-function form JS

Combine this with Lesson 3 and you get the form you'll see most often in real code — an async arrow function assigned to a const, often exported from a module: 

import { Task } from "./types";

export const fetchTasks = async (): Promise<Task[]> => {
  const response = await fetch("https://api.example.com/tasks");
  const tasks: Task[] = await response.json();
  return tasks.filter(task => !task.done);
};

Read top to bottom: "export an async function fetchTasks that takes no arguments and promises to deliver a Task[]; inside, wait for the network response, wait for it to be parsed as JSON into Task[], then return only the unfinished tasks." Every concept from Lessons 1–3 is here — interface, array type, arrow function, .filter(), module import/export — plus the new async/await/Promise.

5. The trap: forgetting to await

The most common bug — and the thing you'll most often need to spot in AI-generated code — is forgetting await. If you drop it, you're holding the claim ticket instead of the coat: 

// BUG: no await
const tasks = fetchTasks();
console.log(tasks.length);  // undefined — tasks is a Promise, not an array!

// Correct
const tasks = await fetchTasks();
console.log(tasks.length);  // works — tasks is the real array
Good news: this is exactly where TypeScript earns its keep. Without the await, tasks has type Promise<Task[]>, and .length / .filter() aren't valid on a promise — so the type checker complains before you run it. When you see a red squiggle on .something after a function call, "did I forget await?" is the first thing to check.

Try it yourself

Open the TypeScript Playground and paste this (no network needed — delay just simulates work that takes time): 

const delay = (ms: number): Promise<void> =>
  new Promise(resolve => setTimeout(resolve, ms));

const getGreeting = async (name: string): Promise<string> => {
  await delay(500);          // pretend this is a slow API/file read
  return `Hello, ${name}!`;
};

const main = async () => {
  console.log("asking...");
  const message = await getGreeting("Hao");
  console.log(message);
};

main();

Run it — you'll see asking... then, half a second later, Hello, Hao!. Now break it on purpose: delete the await in front of getGreeting("Hao"). TypeScript will flag that message is now a Promise<string>, and if you console.log(message) you'll see a promise object instead of the greeting — the "holding the ticket, not the coat" bug, caught live.
