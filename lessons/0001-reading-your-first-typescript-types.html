Lesson 1 of your TypeScript journey

Reading Your First TypeScript Types

Goal: when AI generates a TypeScript function, you can read it and explain what every part means.

1. JavaScript first: a function with no types

TypeScript is JavaScript plus a type system. Before adding types, here's a plain JavaScript function. It takes a name and an age, and returns a greeting string. 

function greet(name, age) {
  return `Hello ${name}, you are ${age} years old.`;
}

greet("Ava", 30); // "Hello Ava, you are 30 years old."
greet("Ava", "thirty"); // also "works" — JS won't stop you

Nothing stops you from calling greet("Ava", "thirty") with a string instead of a number. JavaScript just runs it — and you might get a bug far away from where the mistake happened.

2. Adding types: the basic syntax

TypeScript lets you write name: Type after a variable, parameter, or return value. Here's the same function with types added: 

function greet(name: string, age: number): string {
  return `Hello ${name}, you are ${age} years old.`;
}
Read it left to right: name: string means "the parameter name must be a string". The : string after the closing ) means "this function must return a string".

Now greet("Ava", "thirty") is a type error — TypeScript catches it before the code even runs, right in your editor.

3. The everyday primitive types

These five cover the vast majority of what you'll see in AI-generated code:

let isActive: boolean = true;
let count: number = 42;
let title: string = "Learning TypeScript";
let tags: string[] = ["ts", "types"]; // array of strings
let nothingYet: undefined = undefined;

let and const are JavaScript keywords for declaring variables (const can't be reassigned, let can). The : type part is the only thing TypeScript adds.

4. Describing objects with interface

AI tools constantly generate interface blocks to describe the "shape" of an object — what properties it has and what type each one is. 

interface User {
  id: number;
  name: string;
  email: string;
  isAdmin: boolean;
}

function printUser(user: User): void {
  console.log(`${user.name} (${user.email})`);
}
Read interface User { ... } as: "Anything labeled User must have these exact properties, with these exact types." : void means "this function doesn't return anything useful."

Quiz: given the User interface above, which call is valid?

5. Union types: "this OR that"

A union type, written with |, means a value can be one of several types. AI output uses this constantly for things like status fields or optional results. 

type Status = "loading" | "success" | "error";

function describe(value: string | number): string {
  return `The value is ${value}`;
}

type Status = "loading" | "success" | "error" creates a custom type alias — Status can ONLY be one of those three exact strings. string | number means "a string, or a number — either is fine."

Quiz: with type Status = "loading" | "success" | "error", which assignment is valid?

6. Optional properties: the ?

A ? after a property name means "this might not be there." Extremely common in AI-generated interfaces for optional fields. 

interface User {
  id: number;
  name: string;
  nickname?: string; // optional — may be undefined
}

const a: User = { id: 1, name: "Ava" }; // OK, no nickname needed
const b: User = { id: 2, name: "Sam", nickname: "S" }; // also OK

Try it yourself

Open the TypeScript Playground and paste the User interface and printUser function from section 4. Try calling printUser with a missing property, or with id: "1" instead of a number. Read the red squiggly error messages — that's TypeScript explaining exactly what's wrong, in the same vocabulary you just learned.
